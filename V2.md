# Finder V2

This document outlines the design decisions and architecture for the second version of the Finder application.

## Current (V1) version limitations

Most of the logic is handled by the components:
- "generator" – it scans the directories, generates HTML files,
- "indexer" – preserves it's own state, pushes updates to Meilisearch.

For some cases, like "missing" files (content referenced by other content), we had to pass some data between these components.

Although the current version of Finder serves its purpose, there are several limitations that have been identified:

- All media handling logic is setup as a chain of GitHub Actions workflows, involving `thumbnailer` custom action, and `media` repository triggering `finder` repository workflow.

## Motivation for refactor

Ideally, we want to have a more modular and maintainable architecture that allows for easier updates and enhancements. Specifically, there are several features we aim to add with the new version:

- Support for OpenGraph images. I.e. generator should know the image corresponding to file (if any) and use it to generate an appropriate image. For optimization, we can cache previously generated OpenGraph images.

- Similar to OpenGraph, support for oEmbed JSON responses to provide rich media previews.

- Since the app is able to run as GitHub Action, it make sence for it to be able to leave file annotations, whenever there is an issue with processing a file or missing config options.

- Main project (info) separated form images (media), i.e. generator expects two directories, therefore walking both directories separately. That requirement may be not ideal for all use cases. For V2 we should _support_ a single directory structure, where media files are located alongside main content files.

## Proposed architecture

- App core: responsible for scanning directories, managing state and coordinating between different "processors".
- Pluggable processors: each processor handles a specific task, such as generating HTML, indexing content, generating OpenGraph images, etc.

In main.go we will initialize the app core, load configuration, and register the necessary processors. The app core will then handle the execution flow, invoking each processor as needed:

```go
a, _ := app.NewApp(...)
a.RegisterProcessor(processors.NewHTMLGenerator(...))
a.RegisterProcessor(processors.NewIndexer(...))

_ = a.Run()
```

Each processor will implement a common interface, allowing the app core to interact with them uniformly:

```go
type Processor interface {
    Process(file File) error
}
```
